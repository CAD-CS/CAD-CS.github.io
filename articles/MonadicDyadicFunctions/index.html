Monadic And Dyadic Functions: Breaking Free From The Mold

Introduction

Due to their early math education or the prevalence of C language-like syntax, developers have grown accustomed to the standard f(x) notation to represent functions. This notation is both intuitive and simple and near universal; used in physics, mathematics, computer science, and everything else related to STEM. There might be some variations, such as foregoing the parenthesis like Haskell, f x, or placing the name inside the parenthesis like the LISP language family, (f x). No matter what the idea is the same, the function called f takes input(s) x and the result will always be the same regardless where the name is located with respect to its arguments.

What if I told you that it doesn't have to be this way? This article will investigate the idea of monadic and dyadic functions in the APL language family and how they break free from the mold.

Monadic Functions

Monadic functions work almost the same as "regular" functions. In the J programming language, a monadic function would look like this: f x. The argument always goes on the right side. This might seem painfully obvious but this does come into play when talking about dyadic functions.

Dyadic Functions

Dyadic functions in J look like this: y f x. At most, a dyadic function can take only two arguments, one on its left and another on its right. This might seem overly restrictive but it is important to realize that languages like APL and J are constructed from the ground up to deal with arrays. Therefore if the developer wishes to feed more than two arguments into the function, they can just treat one of the parameters as an argument vector/list.

One fascinating consequence of this monadic and dyadic split is that a single function, f, can have two different operations depending if it used monadically, f x, or dyadically, y f x.

Take for example, the primitive, %, in the J language. Used monadically, % x, the function yields the reciprocal of the number, 1/x. When used dyadically, y % x, the function yields a simple division, y/x.

This dynamic allows the developer to define a single function that performs two similar operations, thus resulting a decrease in the number of user defined functions.

Conclusion

Despite not being widely used, monadic and dyadic functions offer a glimpse into a unique paradigm that deserves to be researched and explored. They offer unique and elegant solutions which help challenge longstanding norms and unveil new perspectives.
