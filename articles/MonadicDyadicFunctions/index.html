Monads, Forks, And Hooks: Alternative Design Patterns In Non-Object Oriented Languages

Introduction

From the Composite pattern to the Singleton, a wide variety of design patterns have emerged over the decades which, for the purposes of this article, can be broadly broken down into three categories: structural, behavioral, and creational.

All of three of these categories relate directly or indirectly to object-oriented concepts/principles and the patterns themselves rely on OOP tools provided by the languages for their implementation. The composite pattern (structural) is the textbook example of polymorphism and both the observer pattern (behavioral) and the abstract factory pattern (creational) use objects and classes as their building blocks.

Design patterns help tremendously with streamlining and standardizing software development while freeing crucial time for the developer to work and think about and design the novelle components of their software. In other words, the developer doesn't need to reinvent the wheel everytime they embark on a new project. But design patterns aren't unique to OOP as these patterns arise when developers encounter similar classes of problems. Three non-OOP patterns will be investigated, one general pattern from the world of functional programming, and two from the array programming language family.

Monads

The Monad is the functional programming version of a design pattern. The basic idea of a monad is to allow the developer to chain operations. Each of these operations can be linked together with a "bind" operator. This feeds the output of one operation into the input of another. Along the way, the bind operator does extra things to modify the output of the first operation before giving it to the second.

In essence, using monads is just formalized function composition. Each monad would have their own unique bind operator which would suit their needs. Some examples of monads are the failure monad and list monad. The first is used when an operation in a sequence of operations can fail. In the case that does happen, the bind operator returns early without continuing. The second is used when an operation returns multiple results where each result needs to get fed into the next operation. For example, given a single number, n, an operation creates a list of n numbers which would need to get normalized.

Hooks And Forks

The hook and fork pattern are two patterns found in the array programming language family, more specifically APL and its descendants. In essence, these patterns are a special type of function composition supported by the compiler. This makes them language specific design patterns rather than more abstract patterns discussed in the introduction and the monads section.

Using array programming notation, this is what a hook looks like: (u v) y. In more traditional notation, u(y, v(y)). In english, you first apply function v to argument y which is then inputted into function u alongside the original value. For example, the expression, n * (n + 1) can be easily represented in a single expression using the built-in operators found in APL and J without needing to create a user defined function.

A fork is a similar idea: (f g h) y which is equivalent to g(f(y), h(y)). This pattern is perfect if you want to find the mean value of a list.

Conclusion

We see design patterns popping up anywhere there is a healthy ecostystem of developers collaborating and publishing as can be seen from the wealth of patterns that have arisen for OOP and functional and array programming are no different. As these two paradigms gain more traction and popularity, more unique and domain specific patterns will arise.
